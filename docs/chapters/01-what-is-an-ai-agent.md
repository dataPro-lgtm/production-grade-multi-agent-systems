# 第 01 章：先判断是否需要 Agent

我见过不少 Agent 项目从一张令人兴奋的架构图开始：中间是 Supervisor，四周分布着规划、检索、分析和审核 Agent，箭头连向向量库、知识图谱和一排工具。图很完整，真正困难的问题却还没有回答——这里究竟有什么决策必须交给模型？

本书从相反的方向开始。

我们要建设一套名为 CaseOps 的企业理赔协同系统。贯穿十章的任务只有一句话：

> 调查理赔案件 C-102 为什么仍未完成，确认缺失材料和适用规则；如果需要联系客户，只生成通知草稿，不得自动发送。

这句话同时包含事实查询、规则匹配、时间语义、自然语言理解和外部动作，看起来很适合 Agent。可如果因此直接选择 Agent，我们只是用技术名词替代了架构判断。

我更愿意先问：

- 哪些步骤已经可以被精确描述？
- 哪个判断必须根据中间证据改变路径？
- 模型可以提出什么动作，系统又允许它执行什么？
- 如果暂时不用 Agent，业务目标能完成到什么程度？

本章会先运行一个没有大模型的 CaseOps Slice 0，再从它解决不了的问题中推导 Agent 的位置。这样做不是保守，而是在建立一条以后可以比较的基线：当我们引入模型、工具和多 Agent 后，必须证明新增复杂度换来了什么。

## 1. 先把业务问题写成可验证事实

C-102 是一个合成案件，不包含真实客户数据。当前已知事实如下：

| 业务对象 | 固定事实 |
|---|---|
| 案件 | `C-102`，版本 7 |
| 租户 | `tenant-demo` |
| 当前状态 | `waiting_for_documents` |
| 规则 | `motor-claim-standard@2026.1` |
| 已收材料 | 损失情况说明、身份材料 |
| 必需材料 | 损失情况说明、身份材料、事故证明 |
| 动作限制 | 可以生成补件通知草稿，不得直接发送 |

只看这组事实，调查路径并不神秘：

1. 在当前租户内读取案件；
2. 取得案件绑定的规则版本；
3. 确认该版本在案件提交日有效；
4. 用必需材料集合减去已收材料集合；
5. 如果存在缺件，生成草稿并等待人工审核。

这里没有哪一步需要开放式规划。下一步不会因为模型“思考得更好”而变化，集合差也不会因为 Prompt 更长而更准确。传统代码足以完成任务，而且更容易证明正确。

CaseOps Slice 0 把这条路径实现为一个确定性应用用例。下面是经过压缩的核心结构；完整代码位于 [`application.py`](https://github.com/dataPro-lgtm/production-grade-multi-agent-caseops/blob/chapter-01-slice-0/src/caseops/application.py)：

```python
case, case_evidence = cases.get(tenant_id, case_id)
policy, policy_evidence = policies.get(
    tenant_id,
    case.policy_id,
    case.policy_version,
)

missing = tuple(
    item
    for item in policy.required_documents
    if item.code not in set(case.received_document_codes)
)

return InvestigationResult(
    decision=decision_for(missing),
    recommended_action=draft_only_action(missing),
    evidence=(case_evidence, policy_evidence),
)
```

这段代码有三个值得注意的边界。

第一，租户身份进入仓储查询条件，而不是先读出案件再由模型判断能否查看。未授权数据根本不应该进入上下文。

第二，案件与规则都返回带版本和哈希的证据引用。结论不是一段孤立文本，而是可以回到 `case://C-102@7` 与 `policy://motor-claim-standard@2026.1`。

第三，当前动作空间里只有 `draft`，没有 `send`。禁止自动发送不是 Prompt 中的一句提醒，而是代码接口、策略和测试共同形成的事实。

## 2. 一个能运行的确定性基线

本章代码独立维护在 [CaseOps 仓库](https://github.com/dataPro-lgtm/production-grade-multi-agent-caseops)，固定版本为 `chapter-01-slice-0`。读者可以从空环境启动 API 和 PostgreSQL：

```bash
git clone https://github.com/dataPro-lgtm/production-grade-multi-agent-caseops.git
cd production-grade-multi-agent-caseops
git checkout chapter-01-slice-0
docker compose up --build -d
```

发起调查：

```bash
curl --fail-with-body \
  --request POST \
  http://localhost:8080/v1/cases/C-102/investigations \
  --header 'Content-Type: application/json' \
  --header 'X-API-Key: caseops-local-dev-key' \
  --header 'Idempotency-Key: book-ch01-c102-0001' \
  --data '{"notification_action":"draft"}'
```

响应的关键部分如下：

```json
{
  "replayed": false,
  "result": {
    "decision": {
      "code": "MISSING_REQUIRED_DOCUMENTS",
      "explanation": "案件缺少规则要求的必要材料：事故证明。"
    },
    "recommended_action": {
      "type": "draft_notification",
      "execution_policy": "human_approval_required",
      "side_effect": "none"
    },
    "evidence": [
      {"ref": "case://C-102@7"},
      {"ref": "policy://motor-claim-standard@2026.1"}
    ]
  }
}
```

再次使用相同幂等键请求，服务返回同一个 `investigation_id`，`replayed` 变为 `true`；调查记录、审计事件和 Outbox 事件都不会重复写入。

到这里，我们已经完成了原始任务中结构化、确定性的一部分。系统不是 Agent，但它并不因此低级。它有版本化 API、事务、迁移、租户边界、幂等、审计、指标和自动化测试；这些能力以后仍会保留在 Agent 之外。

这也是我对“生产级”一词的使用方式：我只用它描述已经存在、能够验证的工程属性。Slice 0 是生产导向的参考实现，但还没有接入真实企业身份、容量目标、备份恢复和合规控制，所以我不会声称它已经替某家企业完成生产认证。

## 3. 什么时候确定性方案开始不够用

如果案件资料始终结构化，规则始终明确，路径始终固定，CaseOps 可以一直保持工作流。真正推动 Agent 出现的，不是“我们想使用大模型”，而是需求发生了变化。

设想下一批案件出现以下情况：

- 客户上传了一份没有标准材料编码的扫描件；
- 文档把“事故证明”写成地方机构使用的另一种名称；
- 新旧规则同时命中，必须根据发生地和事故日期选择；
- 一个数据源显示材料已收到，另一个数据源仍显示缺失；
- 调查过程中发现案件可能重复提交，需要转向关联案件取证；
- 证据不足时，系统必须决定继续检索、请求澄清还是停止。

这时路径不再能被一张固定流程图完整描述。系统要根据观察结果选择下一步，而且选择可能在运行中改变：

```text
读取案件
  ├─ 材料结构化且规则唯一 → 确定性判断
  ├─ 材料名称不明确       → 文档理解
  ├─ 规则版本冲突         → 补充时间与地域证据
  ├─ 数据源相互矛盾       → 交叉核验
  └─ 证据仍不足           → 请求澄清或转人工
```

这里出现了 Agent 的合理位置：它处理的是**受约束动作空间中的动态路径选择**。身份验证、租户隔离、规则有效期计算、数据库事务和通知发送权限仍不需要模型接管。

我通常把这种设计称为“确定性主干上的 Agentic 岛屿”。灵活判断被放在局部，业务真相和副作用控制仍由可测试代码掌握。

## 4. Agent 的工程定义

大模型本身不是 Agent。它接收输入并生成输出，不会自动拥有目标、状态、工具或权限。工具调用也不是充分条件：如果程序总是按固定顺序调用三个工具，模型即使参与参数生成，整体仍然是工作流。

从工程角度看，Agent 至少要形成一个持续闭环。

![Agent 围绕目标运行的最小闭环](../assets/images/chapter-01/agent-minimal-loop.png)

*图 1-1　观察结果会改变下一动作，运行时则负责限制动作空间、执行动作和保存状态。*

我在本书中采用如下定义：

> AI Agent 是一个围绕目标运行，能够观察任务与环境状态，在受约束的动作空间中选择下一行动，根据行动结果更新状态，并持续决定继续、停止、拒绝或转交的系统。

这一定义包含五个可以落实到代码的部分。

**目标**不是“你是一名理赔专家”这样的角色描述，而是包含范围、成功标准、约束和终止条件的任务契约。C-102 的成功标准不是“给出专业分析”，而是确认缺失材料与适用规则，结论带证据，通知只生成草稿。

**观察**不只是用户最后一句话。它还包括当前任务状态、已用预算、工具结果、错误类型、权限决策和证据缺口。

**决策**是从允许的候选动作中选择下一步。生产系统需要结构化动作提议，不需要把不可验证的长篇“思维过程”保存为控制依据。

**行动**是系统真正读取或改变环境的地方。模型可以提出调用意图，运行时必须重新校验身份、权限、参数、风险和幂等语义。

**状态更新与终止**决定系统能否恢复和停止。完成、证据不足、超时、拒绝、等待审批和转人工都应成为显式状态，而不是藏在对话文字中。

Agent 性因此不是由模型数量、Prompt 长度或工具数量决定的。关键在于：模型是否根据反馈参与了下一步路径控制。

## 5. 不要把两种自治混在一起

“这个 Agent 自治程度很高”通常是一句信息不足的话。我会把自治拆成两个坐标：

- **规划自治**：模型可以在多大范围内决定下一步做什么；
- **执行自治**：系统允许它在多大风险范围内真正改变外部世界。

![规划自治与执行自治的二维边界](../assets/images/chapter-01/autonomy-matrix.png)

*图 1-2　规划空间可以较大，执行权限仍然可以很低；两条轴必须分别设计。*

这个区分能解释许多看似矛盾的系统。

研究型 Agent 可以自由调整检索路径，拥有较高规划自治，但只能读取公开资料，执行自治很低。反过来，固定退款流程的路径完全由代码决定，规划自治为零，却可能执行真实资金操作。后者不是 Agent，风险也绝不低。

CaseOps 的目标位置是“较高规划自治、较低执行自治”：未来的调查 Agent 可以决定查哪份材料、何时交叉核验、是否请求澄清；涉及发送通知、修改案件状态或触发赔付时，必须进入确定性策略与审批链。

这比单一的 L0—L5 等级更接近真实架构，因为它不会把“更会规划”误写成“应该拥有更多权限”。

## 6. 模型负责模糊判断，运行时负责不可让步的边界

一个生产 Agent 不能被简化为“模型 + Prompt + 工具”。当模型参与路径控制后，系统至少还要守住以下责任：

| 责任 | 可以让模型参与什么 | 必须由确定性系统完成什么 |
|---|---|---|
| 目标 | 解释自然语言、提出子目标 | 固定范围、预算、成功和终止条件 |
| 动作 | 从候选动作中选择、生成参数草案 | 鉴权、授权、Schema 校验、实际执行 |
| 状态 | 总结观察、建议下一状态 | 持久化、版本控制、合法状态转换 |
| 证据 | 判断相关性、识别冲突 | 来源、时间、哈希、访问范围 |
| 副作用 | 说明为什么需要动作 | 幂等、事务、审批、回滚与对账 |

这张表表达的是责任分配，不是技术选型。小系统可以把这些责任放在一个进程中，大系统也可以拆成状态服务、策略引擎和工具网关。重要的是责任真实存在，并有独立测试。

有一条边界尤其不能含糊：

> 模型可以提出动作，但不能给自己授权。

无论模型多确信某封通知应该发送，它都不能把“业务上看起来合理”转换成“系统已经授权”。工具返回、检索文档和其他 Agent 的消息同样是外部输入，可能错误、过期或包含提示注入。它们进入决策上下文之前必须带来源和信任标记。

[OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) 将目标劫持、工具滥用、身份与权限滥用、Agent 通信投毒等风险放在完整行动链中讨论。它强化了一个重要认识：安全目标不是让模型永远不受骗，而是即使模型判断失误，确定性边界仍能限制影响范围。

## 7. 多个角色还不等于多 Agent

如果系统依次使用“材料分析员”“规则专家”“审核员”三个 Prompt，但它们共享同一状态、同一权限、同一故障域，并始终按固定顺序执行，我更愿意称它为多阶段 LLM 工作流。

有价值的 Agent 拆分需要真实边界。

![判断 Agent 是否值得拆分的工程边界](../assets/images/chapter-01/engineering-boundaries.png)

*图 1-3　领域、数据、工具、权限、状态、故障和所有权，决定拆分是否成立。*

我会逐项追问：

- 是否拥有不同的业务目标和验收标准？
- 是否只能访问不同的数据或工具？
- 是否需要独立身份和权限？
- 是否拥有独立状态与恢复路径？
- 失败是否应该隔离、降级或单独重试？
- 是否由不同团队独立发布和承担 SLO？
- 是否确实存在可以并行处理的子任务？

如果答案大多是否定的，拆分通常只会增加路由错误、上下文丢失、分布式状态和成本归因问题。两个 Prompt 不会自动形成两个故障域，五次模型调用也不会自动提高可靠性。

CaseOps 最终可能拆出案件调查、规则研究和客户沟通能力，但那是后续章节需要证明的结论，不是第 1 章预设的答案。

## 8. 从最低复杂度开始选择架构

我使用一条简单的架构阶梯。每向上一层走，系统获得更多适应性，也承担更多状态、测试、安全和运维成本。

![从确定性代码逐步升级的架构阶梯](../assets/images/chapter-01/architecture-escalation.png)

*图 1-4　只有下一层无法满足需求，并且收益能够被评测证明时，才增加自治或分布式复杂度。*

**确定性代码**适合规则明确、输入结构稳定的计算和控制。权限判断、状态转换、金额计算、Schema 校验和事务提交都应优先留在这一层。

**普通 LLM 调用**适合摘要、分类、抽取和改写。它处理非结构化信息，但不持续控制路径。

**LLM 增强工作流**把模型放入预定义步骤。局部分支可以存在，整体路径仍由代码控制。许多企业任务停在这里已经足够。

**单 Agent**在受限动作集合中根据反馈改变路径。它适合无法事先枚举完整步骤、但责任仍集中在一个目标中的任务。

**多 Agent**只有在领域、权限、状态、故障或所有权需要独立时才成立。它解决的是分布式责任问题，而不是用更多角色装饰工作流。

这几层不是成熟度排名。一个可靠的确定性服务，通常比一个无法恢复、无法审计的多 Agent 系统更接近生产要求。

## 9. CaseOps 的三种候选方案

把 C-102 放回架构选择中，可以得到三种方案。

![CaseOps 从工作流到多 Agent 的三种候选架构](../assets/images/chapter-01/caseops-architecture-options.png)

*图 1-5　三种方案共享确定性控制面，差异只在动态调查责任放在哪里。*

### 方案 A：确定性调查服务

API 读取结构化案件和规则，应用服务计算缺件，事务写入结果、审计与 Outbox。优点是行为可预测、成本低、易于测试；缺点是不能理解未知材料、处理规则冲突或动态取证。

这正是 Slice 0。对当前固定输入，它是满足需求的最低复杂度方案，所以本章选择它。

### 方案 B：受约束单 Agent

在确定性主干中加入一个 Investigation Agent。它可以使用“读取案件”“解析材料”“查询规则”“交叉核验”“请求澄清”这些只读或无副作用动作，根据证据改变路径。所有工具调用仍通过策略层，通知仍然只生成草稿。

当非结构化材料和动态路径成为真实需求时，这会是下一步优先方案。

### 方案 C：受监督的多 Agent

Supervisor 保留全局目标、预算和结果责任，将规则研究、材料核验等子任务委托给拥有独立工具、权限和状态的 Agent，再按 Join Contract 汇合证据。

只有当单 Agent 已经出现可测量的上下文冲突、权限混杂、独立故障恢复或并行收益时，我才会选择这一方案。第 3 章会让这些压力真实出现，再完成拆分。

## 10. 把架构判断写进 ADR

“先做简单版本”不能只是一句口号。团队需要记录为什么选择、何时升级、怎样回退。CaseOps 的第一份 ADR 固定了四个判断：

1. C-102 当前路径可以确定性描述；
2. 模型尚未提供可验证的必要收益；
3. 通知只能生成草稿，外部副作用不进入当前动作空间；
4. 只有动态路径、非结构化证据或独立工程边界出现后才升级。

完整记录见 CaseOps 的 [ADR-0001](https://github.com/dataPro-lgtm/production-grade-multi-agent-caseops/blob/chapter-01-slice-0/docs/adr/0001-start-with-a-deterministic-modular-monolith.md)。

在自己的项目里，我建议至少回答以下问题：

| 判断问题 | 必须给出的证据 |
|---|---|
| 不使用 Agent 能完成多少目标？ | 可运行基线与未满足需求 |
| 哪个决策必须动态发生？ | 无法由固定分支可靠覆盖的案例 |
| 模型可以选择哪些动作？ | 有限动作清单与参数契约 |
| 哪些动作永远不能自主执行？ | 策略、审批与副作用边界 |
| 状态由谁拥有？ | 状态 Schema、版本与恢复责任 |
| 升级后是否真的更好？ | 同一数据集上的质量、延迟、成本和风险比较 |
| 如果收益不足如何回退？ | 保留的确定性路径与兼容策略 |

如果团队无法回答“哪个决策必须由模型动态完成”，我不会建议继续 Agent 化。如果无法回答“状态和权限由谁拥有”，我更不会建议拆成多 Agent。

## 11. 本章实践与验收

完成本章，不是画出一张包含 Agent 的图，而是生成一组可复核证据。

先运行 CaseOps：

```bash
docker compose up --build -d
make verify
make security
```

然后验证五件事：

1. `/health/ready` 只有在数据库可用时返回 200；
2. C-102 的缺失材料为 `ACCIDENT_CERTIFICATE`；
3. 相同幂等键不会产生重复调查、审计和 Outbox 事件；
4. 其他租户查询 C-102 时得到与未知案件相同的 404；
5. 结果中的建议动作始终是 `side_effect: none`。

最后，为自己的场景写一份 ADR：

- 给出一个不用 Agent 的最小实现；
- 标出它真正无法解决的动态决策；
- 分开定义规划自治和执行自治；
- 列出确定性系统不可让渡的责任；
- 写明升级和回退条件。

本章配套代码已经通过 Ruff、严格 Mypy、单元/集成/契约测试、覆盖率门槛、Bandit、依赖漏洞审计、PostgreSQL 迁移与容器端到端验证。具体结果随代码版本维护在 CaseOps 仓库；不要把本书中的一次验证数字理解为未来所有版本的永久结论。

## 12. 从“是否需要”走向“怎样行动”

现在可以给出本章的结论。

Agent 的工程分界不在于是否使用大模型、是否调用工具，也不在于 Prompt 中出现了几个角色。真正改变系统性质的是：模型是否围绕目标，根据环境反馈参与下一步路径选择。

即便需要 Agent，也不意味着整个系统都要自治。可靠的生产设计通常保留确定性主干，只把无法预先编码、确实需要语义判断的局部交给模型。规划自治与执行自治必须分开设计，模型可以提出动作，但不能给自己授权。

CaseOps Slice 0 因此停在确定性服务：它已经能可靠完成结构化缺件判断，同时诚实暴露了无法理解未知材料、无法处理证据冲突和无法动态取证的限制。

下一章会从这些限制出发，为 CaseOps 加入第一个受约束 Agent。但重点不在 Prompt，而在动作如何进入显式状态机：工具请求怎样校验，超时后怎样恢复，响应丢失时怎样避免重复副作用，以及 MCP 在这条责任链中究竟解决什么问题。
