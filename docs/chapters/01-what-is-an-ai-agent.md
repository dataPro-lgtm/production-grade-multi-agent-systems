# 第 01 章：到底什么才算 AI Agent？

在讨论多智能体架构之前，我们必须先解决一个看似基础、实际上极容易被混淆的问题：**到底什么样的系统才算 Agent？**

如果把“调用过大模型”“能够调用工具”或者“Prompt 里写了角色”当作判断标准，那么几乎所有现代 AI 应用都可以被叫作 Agent。这样做虽然方便传播，却会让架构讨论失去精度。一个摘要程序、一个按固定顺序执行的报表流水线，以及一个能够根据环境反馈改变行动路径的系统，面临的状态管理、测试、安全和运维问题完全不同。把它们放进同一个概念盒子，团队就很难对系统复杂度形成共同判断。

本章将从一个数据分析问题出发，逐步建立一套工程化的判断方法。我们的目标不是为“Agent”寻找一个永远正确的哲学定义，而是回答三个会直接影响系统设计的问题：

1. 系统中的下一步行动究竟由谁决定？
2. 模型被允许在多大的空间内自主决策？
3. 把一个系统拆成多个 Agent，是否对应了真实、可验证的工程边界？

这三个问题构成了全书后续内容的起点。

## 1. 从一个退款率问题开始

假设业务负责人向数据系统提出如下问题：

> 分析本月华东地区付费客户的退款率为什么上涨，主要影响了哪些产品？如果排除新客户，这个趋势还成立吗？

这个请求表面上只是一次指标分析，实际却包含了一串相互依赖的判断：什么是退款率，分母应该使用支付订单还是完成订单；“本月”应该与上月、去年同期还是某个活动前窗口比较；华东地区按用户注册地还是订单履约地计算；新客户如何定义；退款率上涨来自退款订单增加，还是订单总量下降。

系统还需要寻找相关数据表和 Join 路径，生成并校验 SQL，执行查询，判断证据是否充足，再决定是否继续按产品、渠道或客群下钻。最终结论必须能够回到数据证据，而不能只给出一段听起来合理的解释。

面对同一个业务请求，我们可以构造两套行为完全不同的系统。

### 1.1 固定工作流：路径由开发者预先决定

第一套系统按照预先写好的顺序执行：

```python
query_overall_metric()
query_by_region()
query_by_channel()
query_by_product()
query_by_customer_segment()
generate_report()
```

无论第一次查询返回什么，它都会扫描所有维度。大模型可以参与意图抽取、SQL 生成或报告撰写，但它并不控制整体执行路径。系统的分支、顺序和终止条件都由代码预先确定。

这是一套 **LLM 增强工作流**。它可能非常有价值，也可能比 Agent 更适合生产环境，但它的本质仍然是工作流。

### 1.2 动态行动系统：路径由反馈改变

第二套系统只接收目标、约束和可用工具。它先查询总体结果，发现某个产品系列贡献了退款增量的 72%，于是停止扫描低贡献维度，转而检查该产品的新老客户结构。如果查询结果显示样本量过小，它会延长时间窗口；如果指标定义存在歧义，它会暂停并向用户澄清。

它的执行逻辑更接近下面的循环：

```python
while not terminal(state):
    observation = observe(state, environment)
    decision = decide(goal, observation, constraints)
    result = execute(decision.action)
    state = update(state, result)
```

此时，代码仍然负责执行工具、验证参数、检查权限和保存状态，但模型开始参与决定“接下来做什么”。行动结果会反过来改变后续选择。

这两套系统最重要的区别，不是模型数量，也不是工具数量，而是：

> **执行路径是否会根据中间反馈动态改变，以及模型是否拥有选择下一动作的权力。**

这条分界线比“是否调用大模型”更有工程意义，因为一旦路径选择权交给模型，系统的测试空间、安全暴露面和故障模式都会发生变化。

## 2. 从大模型应用到 Agent

大模型本身不是 Agent。模型接收输入并生成输出，它不会自动拥有目标、工具、状态或环境。只有当模型被嵌入一个持续运行的系统，并开始参与工作流控制时，Agent 性才会出现。

为了避免概念混用，可以把常见系统分为四类。

| 系统类型 | 谁控制路径 | 是否根据反馈改变行动 | 典型用途 |
|---|---|---:|---|
| 普通 LLM 应用 | 单次调用或应用代码 | 否 | 摘要、翻译、分类、抽取 |
| LLM 增强工作流 | 开发者预定义 | 局部、有限 | 合同解析、报表流水线、内容审核 |
| 单 Agent | 模型在约束内选择 | 是 | 研究、数据分析、代码修复 |
| 多 Agent 系统 | 多个决策单元协作 | 是 | 跨领域客服、复杂研究、软件工程 |

这四类系统不是能力排行榜。固定工作流并不低级，多 Agent 也不天然先进。它们只是把不确定性放在了不同位置。

### 2.1 Agent 的最小闭环

从工程角度看，一个 Agent 至少要形成“观察—决策—行动—更新”的闭环。

![AI Agent 的最小运行闭环](../assets/images/chapter-01/agent-minimal-loop.png)

*图 1-1　Agent 围绕目标持续观察、决策、行动并更新状态，直到完成、拒绝、失败或转交人工。*

**观察（Observe）** 不只是读取用户的一句话。Agent 需要看到当前目标、已经完成的步骤、工具返回、错误类型、剩余预算、权限状态以及环境中的新事实。观察决定了模型在什么信息基础上行动。

**决策（Decide）** 是从允许的候选动作中选择下一步。生产系统不需要模型输出无法验证的“神秘思维过程”，而需要结构化的决策结果，例如动作名称、参数、依赖证据、风险等级和是否继续。

```json
{
  "action": "query_refund_by_product",
  "arguments": {
    "region": "华东",
    "period": "2026-07"
  },
  "reason_code": "REGIONAL_INCREASE_CONFIRMED",
  "risk_level": "read_only",
  "required_evidence": [
    "refund_count",
    "paid_order_count"
  ]
}
```

**行动（Act）** 是系统真正影响环境的时刻。查询数据库、创建文件、更新工单和发送消息都属于行动。模型可以提出行动意图，但实际执行必须经过确定性控制层。

**更新（Update）** 将执行结果转化为新的任务状态。工具调用成功、临时超时、参数错误、权限拒绝、证据不足和等待审批，应该成为可持久化、可恢复的状态，而不是只留在自然语言对话里。

因此，本书采用如下定义：

> **AI Agent 是一个围绕目标运行，能够观察自身与环境状态，在受约束的动作空间内选择并执行行动，根据结果更新状态，并持续决定下一步直到进入终止状态的系统。**

这里的“终止”与“继续”同样重要。不能可靠停止的系统，不是更自主，而是失去控制。

### 2.2 工具不是 Agent 性的充分条件

工具让模型从“生成文本”走向“影响世界”，但一次工具调用并不会自动构成 Agent。

一个程序可以先让模型生成 SQL，再由代码执行 SQL，最后让模型总结结果。如果顺序固定、失败分支固定、模型不能根据结果改变下一步，那么它仍然是工作流。

反过来，一个 Agent 也不必拥有几十个工具。只要系统围绕目标持续运行，并能根据反馈在一个有限动作集合中调整路径，即使只有检索、计算和请求澄清三个动作，它仍然具有 Agent 性。

工具改变的是能力边界；路径控制权决定的是系统性质。

## 3. 一个生产级 Agent 由什么构成

把 Agent 简化为“模型 + 工具 + Prompt”有助于快速入门，却不足以指导生产设计。真正上线时，系统至少需要同时回答五组问题。

| 工程层 | 必须回答的问题 | 典型产物 |
|---|---|---|
| 目标层 | 为什么运行，何时算完成？ | 成功标准、范围、约束、终止条件 |
| 决策层 | 下一步做什么，依据是什么？ | 计划、路由、动作选择、证据要求 |
| 行动层 | 如何读取或改变环境？ | Tool Schema、参数校验、幂等与回滚 |
| 状态层 | 当前做到哪里，哪些事实有效？ | State、Checkpoint、记忆与知识引用 |
| 控制层 | 什么可以做，失败后怎么办？ | 权限、审批、预算、重试、评测与审计 |

这五层并不是五个独立服务，而是五种不能被遗漏的责任。小型 Agent 可以把它们放在同一个进程里；大型系统可以将其拆成状态服务、工具网关、策略引擎和评测平台。重要的是责任必须存在，并且能够被测试。

### 3.1 角色不等于目标

“你是一名资深数据分析师”只定义了表达和行为倾向，没有说明系统应该交付什么，也没有说明什么时候停止。

一个可执行目标至少应包含：

- **目标对象**：分析哪项业务问题；
- **范围**：时间、地区、客户和数据边界；
- **成功标准**：怎样才算形成可靠结论；
- **约束**：预算、权限、禁止动作和证据要求；
- **终止条件**：完成、拒绝、超时、转人工或证据不足。

例如：

> 分析指定时间段内退款率变化，识别贡献最高的驱动因素。所有结论必须引用可追溯查询结果；不得访问未授权数据；当分母定义不明确或证据不足时，必须请求澄清或明确拒答。

这段定义比任何人格描述都更接近可执行契约。

### 3.2 状态、记忆和知识不是同一个东西

很多早期 Agent 把所有信息都追加到对话历史中。这样做看似简单，却会让执行状态、用户偏好和领域事实相互污染。

| 信息类型 | 回答的问题 | 生命周期 | 适合载体 |
|---|---|---|---|
| 任务状态 | 当前进行到哪一步？ | 单次任务 | 结构化 State、Checkpoint |
| 对话记忆 | 用户此前表达过什么限制？ | 会话或跨会话 | 消息历史、摘要、记忆库 |
| 领域知识 | 指标和业务规则是什么？ | 跨任务共享 | 数据库、知识图谱、向量库 |
| 决策轨迹 | 为什么选择这个动作？ | 审计周期 | Trace、Audit Log、Context Graph |

任务恢复依赖状态，个性化依赖记忆，事实正确性依赖知识，故障诊断依赖轨迹。它们可以互相引用，但不应该被当成一段无限增长的 Prompt。

### 3.3 模型可以提出动作，但不能授予权限

Agent 最容易被高估的地方，是把“模型判断某件事合理”误认为“系统允许它执行”。

模型可以判断应该重新处理一笔退款，但是否允许重处理，必须由用户身份、业务规则和审批状态决定。模型可以生成 SQL，但是否允许查询某张表、某些行或某些字段，必须由数据权限系统决定。

生产系统需要一条明确边界：

> **模型负责处理模糊判断；确定性系统负责身份、权限、参数合法性、事务一致性和实际执行。**

工具输出、检索文档和其他 Agent 消息也不能被默认信任。它们都是外部输入，可能包含错误数据、过期规则或间接 Prompt Injection。进入决策上下文前，应经过来源标识、内容隔离和必要的安全检查。

## 4. 自治不是开关，而是一条连续谱

系统不是只有“自主”和“不自主”两种状态。自治表现为模型在多大范围内决定动作、持续多久、是否需要审批，以及是否能够改变外部状态。

![从确定性程序到长期自主系统的自治等级](../assets/images/chapter-01/autonomy-levels.png)

*图 1-2　L0—L5 自治连续谱。生产系统通常优先寻找满足目标所需的最低自治等级。*

| 等级 | 模型承担的职责 | 确定性系统承担的职责 | 常见场景 |
|---|---|---|---|
| L0 固定规则 | 无 | 全部逻辑 | 结算、权限判定、规则计算 |
| L1 局部生成 | 抽取、分类、改写 | 编排全过程 | 文档和内容流水线 |
| L2 受控选择 | 在候选动作中路由 | 候选集合、校验、执行 | 工具选择、客服路由 |
| L3 动态规划 | 规划并根据反馈调整步骤 | 权限、预算、状态、工具执行 | 研究、数据分析、代码任务 |
| L4 多 Agent 协作 | 委托、协商、并行处理 | 协议、隔离、仲裁、全局控制 | 跨领域复杂任务 |
| L5 长期自主 | 长时间持续运行和主动触发 | 治理、监控、资源与责任边界 | 少量高度受控场景 |

自治上升会增加适应性，也会扩大：

- 可选动作数量；
- 可能执行的路径数量；
- 需要覆盖的测试空间；
- 工具误用和越权的风险；
- 成本、延迟与责任归因难度。

因此，架构设计的目标不是追求最高自治，而是寻找**完成任务所需的最低自治**。

### 4.1 受约束自治

企业真正需要的通常不是“模型想做什么就做什么”，而是受约束自治：

- 模型理解非结构化意图并生成候选计划；
- Schema 限制计划和工具参数的形状；
- 策略层检查身份、权限、风险和审批；
- 工具层保证幂等、事务、限流和回滚；
- 状态机处理步骤推进、超时、重试和恢复；
- 评测系统判断结果是否达到发布门槛；
- 人在高风险、低置信度或异常情况下接管。

这种设计不是削弱 Agent，而是把模型的灵活性放在它真正擅长的位置。

## 5. 多个角色为什么不等于多 Agent

假设一个写作系统依次使用“研究员”“作者”“审核员”三个 Prompt。三个角色使用同一个模型、相同数据、相同权限，共享一段对话历史，并且总是按照固定顺序执行。

从实现角度看，这更像一个多阶段 LLM 工作流，而不是多 Agent 系统。角色名称改变了 Prompt 的组织方式，却没有创造新的责任边界。

本书把多 Agent 系统定义为：

> **由多个具有独立目标或子目标、独立状态和受限能力边界的决策单元组成，并通过明确协议进行委托、协作或协调的系统。**

这一定义强调“独立决策单元”，而不是“调用了几次模型”。

### 5.1 有价值的拆分来自真实工程边界

判断是否应该拆分 Agent，不要先给角色起名字，而要先寻找边界。

![判断 Agent 是否应该拆分的真实工程边界](../assets/images/chapter-01/engineering-boundaries.png)

*图 1-3　领域、数据、工具、权限、状态、故障、规模和所有权，是比角色名称更可靠的拆分依据。*

| 边界 | 需要回答的问题 | 可验证产物 |
|---|---|---|
| 领域 | 是否拥有独立术语、规则和成功标准？ | 领域模型、能力清单 |
| 数据 | 可以访问哪些表、文档和实体？ | Data Access Policy |
| 工具 | 可以执行哪些读写动作？ | Tool Registry、Schema |
| 权限 | 谁能委托，哪些动作需要审批？ | RBAC / ABAC 策略 |
| 状态 | 谁创建和更新任务状态？ | State Contract |
| 故障 | 失败是否应该独立重试或降级？ | Failure Contract、Runbook |
| 规模 | 是否需要独立扩缩容和成本预算？ | SLO、容量与预算 |
| 所有权 | 是否由不同团队独立发布和运维？ | Owner、Release Policy |

如果两个所谓 Agent 在这些维度上没有明显差异，那么它们很可能不应该成为独立 Agent。

### 5.2 多 Agent 的收益不是免费的

原书强调了专业化、模块化、并行、扩缩容和故障隔离等优势。这些收益确实存在，但只有在系统真正建立了相应边界时才成立。

例如，给两个 Agent 不同的 Prompt，不会自动获得故障隔离；只有当它们拥有独立状态、超时、重试和降级策略时，故障才可能被隔离。把任务分给多个 Agent，也不会自动提高吞吐；如果它们争用同一个模型配额、数据库连接或工具锁，反而可能更慢。

每项收益都对应一项新增成本。

| 期望收益 | 成立条件 | 新增代价 |
|---|---|---|
| 专业化 | 独立知识、工具或评测标准 | 路由错误、跨域信息损失 |
| 并行处理 | 子任务真正独立 | 汇总、去重、竞态和资源争用 |
| 独立扩缩容 | 服务和状态可以分离 | 部署、监控和容量管理 |
| 故障隔离 | 独立超时、重试和降级 | 分布式故障与级联失败 |
| 团队自治 | 清晰接口和所有权 | 版本兼容与协调成本 |

多 Agent 架构的正确问题不是“它能带来哪些好处”，而是“这些好处是否足以覆盖新增的分布式系统成本”。

## 6. 选择满足需求的最简单架构

Agent 系统通常不是一次设计出来的，而是从简单架构逐步演进。

![从确定性代码逐级升级到多 Agent 的架构阶梯](../assets/images/chapter-01/architecture-escalation.png)

*图 1-4　只有当下一层无法满足需求，并且收益可以被评测证明时，才继续增加自治和分布式复杂度。*

### 6.1 确定性代码

当输入、规则、步骤和结果都可以明确描述时，传统代码仍然是最佳选择。税费计算、访问控制、Schema 校验、事务提交和合规阈值不需要模型自由发挥。

### 6.2 普通 LLM 应用

当任务只是对非结构化内容进行翻译、摘要、分类或抽取，一次或少量模型调用通常足够。系统不需要长期目标，也不需要持续行动。

### 6.3 LLM 增强工作流

当整体路径稳定，但某些节点需要语义理解时，把模型放入确定性工作流。开发者控制顺序和分支，模型只处理局部模糊性。这类系统通常最容易测试和治理。

### 6.4 单 Agent

当中间结果会显著改变下一步，开发者无法合理枚举所有路径，并且模型需要在多个工具之间动态选择时，单 Agent 开始有价值。

单 Agent 应当是大多数 Agent 项目的默认起点。它状态集中、调用链短、延迟较低，也更容易观察模型为什么选择某个工具。

### 6.5 多 Agent

只有当单 Agent 已经因真实领域冲突、权限隔离、上下文膨胀、独立扩缩容或故障边界而受限时，才考虑多 Agent。复杂任务本身不是理由；**不可由单一责任单元安全承载的复杂性**才是理由。

## 7. Data Agent：同一个需求的三种架构

回到本章开头的退款率问题。系统需要理解指标、检索元数据、生成 SQL、校验权限、执行查询、分析驱动因素并形成证据化报告。

这套需求至少有三种合理实现。

![Data Agent 的三种候选架构](../assets/images/chapter-01/data-agent-architectures.png)

*图 1-5　单体 Agent、确定性数据工作流与受约束多 Agent 并不是优劣排序，而是不同的复杂度选择。*

### 7.1 方案 A：单体 Data Agent

一个 Agent 持有元数据检索、SQL 生成、只读执行、统计计算、图表和报告工具。它根据查询结果决定是否继续下钻。

这种架构开发快，状态集中，适合单一数据域、只读分析和工具数量有限的早期系统。它的主要风险是工具不断增加后，Prompt 和 Schema 迅速膨胀，工具选择准确率下降，同时权限边界难以表达。

### 7.2 方案 B：确定性数据工作流

系统按照固定管道执行：

```text
意图与参数抽取
→ 指标和维度解析
→ 元数据检索
→ NL2SQL
→ SQL AST 与权限校验
→ 只读执行
→ 驱动分析
→ 证据化报告
```

模型只参与意图理解、SQL 候选生成和自然语言总结。其余节点由代码控制。

对于经营报表、指标问答和固定分析模板，这往往是最可靠的方案。它容易测试、审计和控制成本，缺点是面对开放式下钻时适应性较弱。

### 7.3 方案 C：受约束的多 Agent

当系统跨越多个独立数据域，或者 SQL 生成、分析决策和证据核验拥有明显不同的权限与评测标准时，可以拆成多个责任单元。

| Agent | 主要职责 | 允许能力 | 明确禁止 |
|---|---|---|---|
| Metadata Agent | 解析指标、字段和 Join 路径 | 元数据与 Schema Graph | 执行生产 SQL |
| SQL Agent | 生成和修正查询 | 只读 SQL 执行器 | 绕过 AST 与权限校验 |
| Analysis Agent | 决定下钻方向并计算贡献 | 统计工具、受限结果查询 | 任意访问原始表 |
| Evidence Agent | 核验时间、分母、过滤与结论 | 证据读取、规则校验 | 创造无证据事实 |

这里的拆分价值来自工具、权限、状态和评测边界，而不是四个角色听起来更像一个团队。

### 7.4 推荐演进路线

一个稳健的演进顺序通常是：

1. 先用确定性工作流做稳指标、元数据、SQL、安全和证据契约；
2. 只在动态下钻环节引入一个受约束的 Analysis Agent；
3. 当工具选择或上下文冲突成为可测量瓶颈时，再拆分 Metadata 与 SQL 责任；
4. 当跨域并行、独立权限、独立扩缩容或组织所有权出现后，再升级为 Supervisor 式多 Agent；
5. 每次升级都用评测证明质量收益覆盖延迟、成本和维护复杂度。

这种路线允许系统随需求增长，而不是一开始就承担完整的分布式 Agent 成本。

## 8. 一套可重复使用的架构判断方法

架构决策不应该依赖“这个场景听起来很复杂”。可以从八个维度分别打 0—2 分，迫使团队把复杂性说清楚。

| 判断维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 路径动态性 | 完全固定 | 少量分支 | 依赖反馈持续变化 |
| 领域数量 | 单一领域 | 两个相关领域 | 多个独立领域 |
| 工具空间 | 少且同质 | 数量中等 | 多且异质 |
| 权限差异 | 无 | 部分不同 | 明确隔离 |
| 上下文冲突 | 很低 | 可以分区 | 已影响质量 |
| 并行价值 | 无 | 局部可并行 | 大量独立子任务 |
| 故障边界 | 共享 | 部分隔离 | 必须独立恢复 |
| 所有权与扩缩容 | 单一 | 部分独立 | 多团队或独立容量 |

分数不是数学真理，只是讨论工具：

- **0—4 分**：优先确定性代码或普通 LLM 应用；
- **5—8 分**：优先 LLM 增强工作流；
- **9—12 分**：评估单 Agent 或混合架构；
- **13—16 分**：多 Agent 可能有价值，但仍需数据证明。

在正式 ADR 中，还应记录以下问题：

1. 不使用 Agent，能否完成目标？
2. 哪个决策点必须由模型动态判断？
3. 模型的选择空间是否已经明确限制？
4. 权限是否由模型之外的系统强制执行？
5. 状态能否持久化并从中断处恢复？
6. 错误能否定位到计划、路由、工具或汇总层？
7. 是否存在能够比较架构方案的 Golden Dataset？
8. 如果收益不达标，如何退回更简单的架构？

## 9. 五个常见误判

### 9.1 “会调用工具就是 Agent”

工具调用只说明模型能够请求外部能力。如果所有工具及其顺序由代码固定，系统仍然是工作流。

### 9.2 “多个角色就是多 Agent”

角色是 Prompt 组织方式。只有当角色对应独立决策、状态、能力或责任边界时，拆分才具有架构意义。

### 9.3 “有长期记忆才算 Agent”

Agent 必须维护足够的任务状态，但不一定需要跨会话长期记忆。短任务可以只使用当前状态和 Checkpoint。盲目增加长期记忆还会引入隐私、污染和过期问题。

### 9.4 “自治越高，系统越先进”

自治扩大了能力，也扩大了风险和测试空间。合理自治是满足目标所需的最低自治，不是模型能够承担的最高自治。

### 9.5 “拆成多个 Agent 就能获得可靠性”

可靠性来自超时、重试、幂等、隔离、回退、评测和运维设计。把一次模型调用变成五次模型调用，默认只会增加失败点。

## 10. 本章结论

判断一个系统是否具有 Agent 性，关键不在于它是否使用大模型、是否调用工具、是否拥有记忆，也不在于 Prompt 中出现了几个角色。

真正重要的是：系统是否围绕目标形成持续行动闭环，模型是否能够根据环境反馈改变下一步，以及这种自主决策是否被限制在可观察、可测试、可授权的范围内。

多 Agent 架构则进一步要求多个决策单元之间存在真实边界。领域、数据、工具、权限、状态、故障、规模和所有权，比“研究员”“规划师”“审核员”之类的角色名称更能解释为什么要拆分。

因此，可以把本章压缩为两条原则：

> **完成业务目标所需的最低自主性，才是合理自主性。**

> **能够被真实工程边界解释的 Agent 拆分，才是有价值的多 Agent 架构。**

下一章将进入 Agent 的运行机制：模型如何形成计划、选择工具、处理工具结果，并根据反馈决定继续、重试、澄清、转人工或终止。
