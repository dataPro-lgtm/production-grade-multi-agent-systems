# 《生产级多智能体系统》2.0 出版级重构规划

> 状态：编辑规划稿  
> 适用范围：全书十章、贯穿案例、示例工程、图表、工程模板、参考资料与发布产物  
> 本轮目标：先统一全书的叙事、技术边界和验收标准，再分阶段重写正文

## 一、2.0 要交付的不是“更长的 1.0”

1.0 已经建立了较完整的知识骨架：从 Agent 边界、工具调用和多 Agent 协作，一直延伸到 Context Engineering、生产基础设施、安全、系统合龙、评测、AgentOps 和开源交付。它的主要问题不是知识点太少，而是**知识点密度高于叙事密度，架构结论多于推导过程，章节产物多于可连续运行的实践主线**。

2.0 的目标不是继续增加概念，而是完成四次升级：

1. **从课程讲义升级为技术专著**：让概念在问题、判断、实现、失败和复盘中自然展开，不再依赖大量并列清单。
2. **从十个主题升级为一条系统演进主线**：十章共同推动同一个企业系统从需求判断走向公开交付。
3. **从“告诉读者应该做什么”升级为“带读者做出工程判断”**：给出约束、选择、反例、代价和验证证据。
4. **从技术时点升级为可持续维护的知识体系**：稳定原理与版本化协议分层表达，避免框架 API 变化拖垮正文。

2.0 完成后，读者应当能够独立回答并实践以下问题：

- 什么时候不应该使用 Agent，什么时候单 Agent 已经足够；
- 怎样把模型的动作提议变成安全、可恢复、可审计的执行；
- 怎样按责任、状态、权限和失败边界拆分多 Agent；
- 怎样为模型构造有来源、有权限、有时间语义的 Context；
- 怎样把系统部署成可服务、可观测、可恢复的生产平台；
- 怎样在模型已被误导的前提下仍守住工具、数据和副作用；
- 怎样贯通 Supervisor、A2A、MCP、状态、证据和长任务；
- 怎样用 Golden Dataset、轨迹评测和重复运行证明质量；
- 怎样诊断线上错误、控制成本、恢复状态并治理漂移；
- 怎样让第三方在干净环境中复现、验收和继续维护项目。

## 二、1.0 编辑审计

### 1. 已经具备的优势

- 技术判断比常见 Agent 入门书更严格，明确区分工作流、单 Agent 和多 Agent。
- 始终强调模型不拥有权限，工具执行、安全策略和副作用控制必须位于确定性运行时。
- 已经形成 Tool Contract、Context Pack、Delegation Contract、Evidence Package 等可复用工程对象。
- 第 6—10 章对安全、评测、运营和交付的覆盖明显强于只讲框架调用的同类内容。
- 每章已有架构图、表格、模板和参考资料，可作为 2.0 的研究底稿。

### 2. 必须解决的结构问题

当前十章正文约 22 万字符，包含约 177 个二级标题和 347 个三级标题。信息覆盖很广，但阅读体验容易变成“连续查看设计清单”。

主要问题包括：

1. **案例主线漂移**  
   第 1—2 章使用退款率分析，第 3—6 章转向支付事故，第 7—10 章才进入 C-102 理赔调查。第 10 章却把 C-102 描述成贯穿前九章的系统，叙事事实并不完全一致。

2. **标题颗粒度过细**  
   多数章节包含 18—30 个二级标题，读者不断切换上下文，很难形成连续的推理链。2.0 应把每章压缩为约 8—12 个主要章节，让三级标题只承担必要的内部结构。

3. **“结论先行”过多，“判断过程”不足**  
   1.0 有大量正确结论，但部分结论从定义直接跳到规则，缺少真实约束下的取舍过程、失败路径和替代方案。

4. **工具箱式表达偏重**  
   表格、清单、代码框和金句密度较高。它们适合查阅，却削弱了书籍应有的叙事节奏和作者声音。

5. **实战产物尚未形成一个可运行系统**  
   当前 `examples/` 主要承接第一章，其他章节的代码更多是局部示意。读者无法从同一个仓库逐章运行系统、注入故障并查看证据。

6. **技术先进性分布不均**  
   第 6—10 章对当前协议和生产实践较新，第 1—5 章的案例、代码深度和验证闭环相对薄弱。2.0 需要让前半本书也达到同一专业水位。

7. **作者声音尚未稳定**  
   当前主要使用“我们”和规范性语气，较少呈现作者怎样看问题、为什么警惕某种设计、什么时候愿意接受折中。专业感来自判断的可信度，不来自命令式句子的数量。

### 3. 需要控制的“AI 味”

2.0 不采用机械的同义改写，而从写作结构上消除 AI 味：

- 减少反复使用“不是……而是……”的对偶句；
- 减少每一节都以三条、五条、八条清单收束；
- 避免所有章节使用同一种开场、同一种小结和同一种金句格式；
- 不用抽象形容词代替证据，例如“强大”“完善”“生产级”“智能”；
- 不把公认常识包装成作者洞见；
- 不虚构客户经历、线上指标或“我曾在某项目中”的故事；
- 示例数字必须标明是演示数据、实测数据还是外部研究结果；
- 技术结论必须说明成立条件和不适用边界。

## 三、全书统一叙事

### 1. 三个部分

#### 第一部分：决定系统如何行动

- 第 1 章：判断是否需要 Agent
- 第 2 章：建立可控制的行动循环
- 第 3 章：决定何时拆成多 Agent

这部分回答“系统该不该自治、自治到哪里、为什么要拆分”。

#### 第二部分：让系统获得可信运行条件

- 第 4 章：构造可信 Context 与证据
- 第 5 章：建立生产平台与恢复能力
- 第 6 章：限制被操纵系统的影响范围

这部分回答“模型依据什么行动、运行环境怎样守住状态、最坏情况下怎样控制损害”。

#### 第三部分：把系统合成、证明并交付

- 第 7 章：贯通 Supervisor、A2A、MCP 与状态
- 第 8 章：建立评测与持续回归
- 第 9 章：建立 AgentOps 运营闭环
- 第 10 章：完成系统验收和开源交付

这部分回答“组件怎样成为系统、系统怎样获得证据、上线后怎样持续可信”。

### 2. 贯穿案例：CaseOps 企业理赔协同系统

2.0 使用一个独立设计的复合案例贯穿十章。案例名称暂定为 **CaseOps**，核心任务是：

> 调查理赔案件 C-102 为什么仍未完成，确认缺失材料和适用规则；如果需要联系客户，只生成通知草稿，不得自动发送。

这个任务适合贯穿全书，因为它同时包含：

- 模糊目标和需要澄清的业务语义；
- 多数据源、知识库和时间有效性；
- 只读调查与高风险写操作；
- 跨团队委托、并行取证和结果汇合；
- 隐私、租户隔离、审批与审计；
- 长任务、失败恢复、证据链和多轮对话；
- 可验证的业务事实和开放式自然语言表达。

案例将按章节逐步演进：

```text
第 1 章：先用确定性工作流解决一部分问题，识别真正需要自治的局部
第 2 章：加入 Tool Contract、状态机、Checkpoint、幂等与 MCP
第 3 章：单 Agent 出现工具歧义、权限混杂和状态冲突，才拆分多 Agent
第 4 章：加入政策、材料、时态与证据型 Context
第 5 章：部署为可服务、可观测、可恢复的平台
第 6 章：注入恶意文档、越权动作和隐私攻击，建立确定性安全边界
第 7 章：通过 Supervisor、A2A 1.0、MCP 和运行时证据模型完成合龙
第 8 章：建立 Golden Dataset、轨迹评测、N-run 和发布门禁
第 9 章：用真实失败场景完成诊断、降级、恢复和成本治理
第 10 章：在干净环境中复现、验收、生成制品并公开发布
```

支付事故、数据分析和企业知识调查不完全删除，但降为“迁移案例”或“侧栏案例”，用于证明方法可以迁移到其他领域，不再与主线争夺叙事位置。

### 3. 第一人称作者声音

2.0 使用克制的第一人称。作者不站在讲台上宣读规则，而是作为有经验的架构师陪读者做判断。

可以使用：

- “我更愿意先问……”
- “这里我会保留确定性主干，只把这个判断交给模型。”
- “如果团队无法回答状态由谁拥有，我不会建议继续拆 Agent。”
- “这个实现能跑，但我不会因此称它为生产就绪。”

不使用：

- 无法证实的客户项目经历；
- 把个人偏好写成行业定律；
- 用“显然”“众所周知”跳过推导；
- 为了显得资深而堆砌产品名。

案例中的组织、数据和事故均标注为复合案例。真实工程经验通过决策质量、失败模式和验证方法体现，而不是依靠不可核验的故事。

## 四、出版级章节写法

每章围绕一个核心工程问题展开，但不强制复制完全相同的模板。通常包含以下构件：

1. **一个需要做决定的现场**：用业务问题或生产事故打开章节。
2. **问题为什么比表面更难**：揭示状态、权限、证据或随机性。
3. **最小模型**：先给读者一个足够工作的心智模型。
4. **从简单方案开始**：先实现最低复杂度版本。
5. **让方案在真实约束下失败**：用失败推动下一层设计。
6. **工程化实现**：给出合同、状态、代码和运行路径。
7. **替代方案与代价**：解释为什么选择它，以及何时不选。
8. **可运行实践**：明确目录、命令、输入和预期证据。
9. **验收与迁移**：告诉读者如何证明完成、如何迁移到自己的系统。
10. **自然过渡到下一章**：留下下一层尚未解决的问题。

### 正文、代码和图表的分工

- 正文负责推导和判断，不复述代码。
- 代码只展示决定系统行为的关键部分，完整实现放入示例工程。
- 表格用于真正需要横向比较的内容，不把连续论述切成单元格。
- 架构图优先表达责任、数据流、控制流和失败传播，不画产品 Logo 集合。
- 每张图只回答一个问题，并在正文中解释怎样阅读。
- 每个代码示例必须标明是伪代码、核心片段还是可直接运行代码。

### 术语和版本策略

正文分为三层：

1. **稳定原理**：控制权、状态所有权、最小权限、幂等、证据、SLO、评测有效性。
2. **协议语义**：MCP、A2A、OpenTelemetry、OAuth、SLSA 等标准对象和责任。
3. **参考实现**：LangGraph、具体模型、图数据库、向量库、容器平台和评测框架。

协议和框架内容必须注明版本或成熟度。草案、实验能力和厂商实践不能写成稳定行业标准。

## 五、十章重构蓝图

## 第 1 章：先判断是否需要 Agent

### 核心问题

在加入工具、记忆和多 Agent 之前，系统究竟需要多少自治？

### 主要重构

- 用 C-102 调查替换退款率作为主线开场。
- 从固定查询工作流开始，让读者看到确定性方案已经能解决什么。
- 再加入需要动态澄清、按证据改变路径和主动停止的部分。
- 把“Agent 定义”从概念争论改为工程判定：目标、观察、动作空间、状态更新和终止。
- 将规划自治与执行自治分成两个坐标，避免把“会计划”误写成“可以行动”。
- 用最小复杂度阶梯比较：普通代码、LLM 调用、工作流、单 Agent、多 Agent。
- 删除重复误区清单，将误区放回相应判断过程。

### 读者产物

- Agent Suitability Canvas
- 自治边界矩阵
- 第一份架构决策记录 ADR-001
- CaseOps Slice 0：确定性理赔调查工作流

### 实战验收

读者能够用同一份需求分别设计工作流、单 Agent 和多 Agent 方案，并解释为什么只选择其中复杂度最低且满足要求的一种。

## 第 2 章：把动作提议变成可靠执行

### 核心问题

模型生成 Tool Call 以后，系统怎样安全地执行、恢复并留下证据？

### 主要重构

- 从一次只读案件查询开始，逐步加入失败、重试和草稿写入。
- 明确 Intent、Authorization、Execution、Observation 和 State Update 五个阶段。
- 将 ReAct 从提示技巧还原为状态机。
- 完整解释 `tool_call_id`、幂等键、外部副作用和未知结果。
- 区分 Checkpoint 与业务事实，避免把恢复点误当成事务保证。
- 使用 MCP 当前正式规范讲清 Host、Client、Server、能力协商、Tools、Resources 和授权边界。
- 增加 OAuth 2.1、Protected Resource Metadata、Token Audience 和任务身份绑定，但不把认证协议写成业务授权。

### 读者产物

- 可执行 Tool Contract
- 显式 Agent State Schema
- 本地 Tool 与 MCP Tool 的等价适配层
- Checkpoint、幂等与副作用对账测试

### 实战验收

在进程崩溃、工具超时和响应丢失三种情况下，系统能够恢复到明确状态，并证明写操作没有被静默重复。

## 第 3 章：在真正需要时拆成多 Agent

### 核心问题

什么工程边界足以抵消多 Agent 引入的协调成本？

### 主要重构

- 先让第 2 章的单 Agent 出现真实瓶颈：工具歧义、上下文污染、权限域混杂和并行状态冲突。
- 用控制权、结果所有权、状态所有权、权限边界和故障边界选择模式。
- 将模式整理为四类：确定性流水线、路由与并行、Handoff、Supervisor/Hierarchy。
- Tree-of-Thoughts、Magentic 和 Consensus 作为推理或元模式讨论，不与部署拓扑混为一类。
- 深入解释 Fan-out/Join、取消、Deadline、部分结果和冲突保留。
- 引入 Delegation Contract、Result Contract 和 Capability Snapshot。
- A2A 在本章只建立语义位置，完整协议实现留到第 7 章。

### 读者产物

- 多 Agent Pattern Decision Record
- Delegation / Result / Join Contract
- 可运行的并行调查与确定性汇合
- 失败传播和取消测试

### 实战验收

读者能够证明每个 Agent 的存在对应真实边界，并能解释删除某个 Agent 后系统会失去什么，而不是只失去一个角色名称。

## 第 4 章：把 Context 做成证据产品

### 核心问题

模型在当前步骤究竟应该看到什么，为什么允许看到，它是否仍然有效？

### 主要重构

- 从 C-102 的错误政策版本和缺失材料开始，而不是先罗列 Context 类型。
- 区分指令、状态、记忆、知识、工具结果和政策。
- 将 Context Pack 写成带来源、权限、时间、版本和用途的运行时数据合同。
- 以 Query Shape 选择 SQL、向量检索、Knowledge Graph 和 GraphRAG。
- 保留 Local、Global、DRIFT 的区别，同时说明它们属于特定 GraphRAG 实现。
- 增加检索内容的间接 Prompt Injection、来源可信度和证据充分性。
- 将压缩改写为“保留不变量和证据”的有损转换问题，并为压缩器建立测试。
- 让 Claim—Evidence 绑定成为回答生成前的结构，不在答案末尾补链接。

### 读者产物

- Source Registry
- Context Pack Schema
- Hybrid Retrieval Router
- Claim—Evidence 图和 Context Trace
- 检索、重排、构包三段式评测

### 实战验收

对于同一个问题，读者能解释每条上下文为什么进入、为什么被拒绝，并能从结论回到有版本的原始证据。

## 第 5 章：建立可服务、可观测、可恢复的平台

### 核心问题

系统“运行着”和系统“能够对外承诺”之间还缺什么？

### 主要重构

- 以 CaseOps 第一次部署失败为叙事主线。
- 从生产保证反推平台，而不是从 Docker、Kubernetes 或数据库产品开始。
- 明确控制面、执行面、数据面、安全面和可观测面。
- 建立 State Ownership Catalog，区分 Source of Truth、Checkpoint、Cache、Index 和 Artifact。
- 深入解释 Startup、Readiness、Liveness 的不同语义和错误配置导致的级联故障。
- 将 Deadline、Timeout、Retry、Backoff、Circuit Breaker 和 Bulkhead 放进一条故障预算链。
- 增加优雅终止、队列 Drain、长任务租约和模型冷启动。
- 用 RPO、RTO 和业务不变量验证恢复，不以“备份成功”作为结论。
- 建立 OpenTelemetry 基础 Trace，但把 GenAI 语义细节留到第 9 章。

### 读者产物

- Production Readiness Contract
- State Ownership Catalog
- 健康探针和流量门禁
- 备份恢复与故障注入脚本
- 最小 SLI/SLO 集

### 实战验收

读者能够模拟服务重启、队列积压、数据库恢复和索引版本不一致，并判断系统是 Healthy、Ready、Degraded 还是 Unsafe。

## 第 6 章：在模型失守后仍限制损害

### 核心问题

如果用户输入、检索内容、工具描述或远端 Agent 已经恶意，系统还剩哪些不可绕过的边界？

### 主要重构

- 保留恶意 Runbook 场景，增加供应链、Memory Poisoning 和跨 Agent Confused Deputy。
- 用 OWASP Agentic Top 10 2026、NIST AI RMF 和当前 MCP/A2A 安全要求校准威胁模型。
- 把 Tool Guard 定义为 Policy Enforcement Point，而不是厂商产品。
- 将授权写成主体、任务、动作、资源、环境和风险的交集。
- 深入解释委托权限只能收缩，接收方必须重新授权。
- 增加 Agent Card 签名、Skill 选择性披露、Tool/Server 版本和发布者信任。
- 将隐私写成从采集、构包、Trace、Memory、缓存到删除传播的完整生命周期。
- 对高风险动作使用绑定参数、版本和有效期的不可变审批。
- 红队从“收集攻击提示词”升级为多次尝试、攻击成功率、影响范围和残余风险评估。

### 读者产物

- Threat Model 与 Trust Boundary 图
- Tool Security Manifest
- 确定性 Policy Engine
- Bound Approval
- 隐私数据流与删除传播测试
- Red-Team Report

### 实战验收

在模型明确产生越权 Tool Call 的情况下，执行仍被确定性边界阻断，并留下足以复盘的安全证据。

## 第 7 章：把组件合成一套分层系统

### 核心问题

目标、权限、状态、任务、证据和错误怎样穿过多个进程后仍保持一致？

### 主要重构

- 继续沿 C-102 完成端到端垂直切片，不在本章重新介绍前六章全部概念。
- 中央 Supervisor 只负责目标分解、依赖和汇合，不直接持有领域数据权限。
- Team Supervisor 将团队目标收敛为命名能力，Worker 负责受控执行。
- 按 A2A 1.0 重写 Agent Card、Interface、Task、Message、Part、Artifact、版本协商和多租户。
- 增加签名 Agent Card、Authenticated Extended Agent Card 和受控能力快照。
- 按 MCP `2025-11-25` 处理能力协商、授权和实验性 Tasks，明确长任务不自动等于 Agent 委托。
- 将 A2A Task State、MCP Task Status 和业务 Result Status 显式映射。
- Context Graph 只定义运行时证据模型，不规定必须使用图数据库。
- 补齐跨 Trace 关联、取消、补偿、部分完成和重放。

### 读者产物

- 可运行的分层 CaseOps
- A2A 1.0 Agent Card 与 Task/Artifact 交互
- MCP Server/Client
- 业务 DAG 与状态映射
- 运行时证据图
- 端到端故障注入报告

### 实战验收

从任意最终 Claim 可以回溯到 Plan、A2A Task、Worker、MCP Tool、身份、权限、数据版本和恢复动作。

## 第 8 章：评估分布、轨迹和业务结果

### 核心问题

当正确答案和正确路径都不唯一时，怎样证明新版本更好？

### 主要重构

- 保留 Oracle、Path 和 Determinism 三个悖论，但用 C-102 的真实评测记录贯穿。
- 四层正确性升级为：结构、行为、语义、推理，并单独处理安全和运营约束。
- Golden Dataset 不存一篇标准作文，而存事实、约束、允许路径、禁用动作和证据引用。
- 对多条有效轨迹定义行为等价类，而不是固定调用序列。
- 分开 Outcome Grader、Trajectory Grader、Policy Grader 和 Human Review。
- LLM-as-a-Judge 必须校准，记录模型、Prompt、温度、弃权和与人工一致性。
- N-run 不只报告平均分，还报告方差、尾部失败和置信区间。
- 对 Agent Hijacking 和高风险失败采用多次尝试评估，避免一次阻断带来的虚假安全感。
- 引入 Baseline vs Candidate 成对比较、污染防护、数据版本和 Eval Report。
- 将 NIST 2026 自动化评测指南按“草案/当前研究”标注，不包装成已定稿标准。

### 读者产物

- Golden Dataset 与确定性 Fixture
- 分层 Grader
- Judge 校准集
- N-run 与行为等价类报告
- 安全对抗评测
- CI Regression Gate

### 实战验收

一次模型、Prompt、Tool 或 Agent Card 变更，能够得到可复核的发布结论，而不是只得到一个脱离样本和误差范围的总分。

## 第 9 章：把 AgentOps 建成生产控制循环

### 核心问题

线上出现“答案看起来对，但路径、成本或证据已经失控”时，团队怎样发现并处理？

### 主要重构

- 以一次 C-102 上下文串扰和工具选择错误组成的复合事故展开。
- 将诊断固定为用户意图、规划、路由、工具、合成五层，不从最后一个 Span 猜根因。
- 使用 Graph、Trace 和 Artifact 三类证据，找到第一个因果错误。
- 对接 OpenTelemetry GenAI 语义约定中的 `invoke_agent`、`execute_tool` 和 Evaluation Event；内容捕获默认关闭并受隐私策略控制。
- 建立目标级 SLI/SLO、错误预算和发布策略，不只看 HTTP 成功率。
- 将重试、降级、熔断、Provider Failover、人工接管和 Kill Switch 写成可审计控制动作。
- 把成本分解到 Goal、Plan、Team、Tool、Model 和失败重试。
- 区分反馈、经验和 Golden Case；生产反馈不能未经审核直接进入 Few-shot。
- 对语义缓存采用风险分层，从 Tool Result Cache 开始，不默认缓存最终答案。
- 增加 Runbook、GameDay 和 Operational Readiness Review。

### 读者产物

- 统一 Trace/Graph/Artifact 关联
- 目标级 SLO 与错误预算
- Incident Bundle
- 运行控制状态机
- 成本归因报告
- GameDay 与 ORR 记录

### 实战验收

读者能够在十分钟内从用户投诉定位到第一个错误节点，执行可回滚止损动作，并判断修复应落在 Prompt、路由、工具、数据、上下文还是基础设施。

## 第 10 章：用独立复现完成系统交付

### 核心问题

怎样证明公开仓库、发布制品和书中的系统是同一个可验证对象？

### 主要重构

- 将前九章产物收敛为 Project Charter、Traceability Matrix 和 Evidence Package。
- 增加 AI 系统影响评估、风险接受和人工监督，参考 NIST AI RMF、ISO/IEC 42001 与 ISO/IEC 42005。
- 用 Vertical Slice 和 Release Gate 控制交付顺序。
- Clean-room Reproduction 必须由未参与开发的人或隔离环境执行。
- 开源仓库补齐 SECURITY、治理、贡献、兼容性、支持窗口和已知限制。
- SBOM 说明“包含什么”，Provenance 说明“怎样构建”，二者不能互相替代。
- 采用 SLSA 1.2 的 Build/Source 思路校准来源证明，并强调 Attestation 必须经过验证才有意义。
- 增加 Secret Scanning、Dependency Review、代码扫描和发布制品签名。

- 系统影响评估
- Evidence Package
- Clean-room Reproduction Report
- SBOM、Provenance 与 Release Manifest

## 六、独立贯穿工程

2.0 的参考实现独立维护在：

`dataPro-lgtm/production-grade-multi-agent-caseops`

书稿与代码采用双仓库结构。书稿负责推导、解释、图表和章节索引；CaseOps 负责运行时、测试、容器、CI、安全扫描和发布。书中不复制完整源码，只引用经过验证的 tag 与 commit。

工程从模块化单体开始，而不是提前建立最终目录。第 1 章已经交付 API、PostgreSQL、迁移、租户边界、幂等、审计、Outbox、日志、指标、容器和 CI；后续只有在章节需求形成真实边界时，才增加 Agent、MCP Server、Context、Eval 或独立服务。

工程遵循五个原则：

1. **主路径可运行**：读者能在普通开发机上启动最小版本。
2. **生产能力可分层启用**：安全、可观测、A2A、Graph 和评测不是启动 Demo 的前置重负。
3. **每章只引入本章所需变化**：通过不可变 Tag 呈现演进，避免读者第一章就面对最终系统。
4. **`main` 始终可运行**：章节版本不是十份分叉代码，修复和新能力沿同一主干演进。
5. **生产级由证据定义**：没有运行、测试、迁移、观测或安全证据的能力，只能写成计划或限制。

## 七、图表与排版重构

2.0 的图表不是对 1.0 图片的批量换色，而是正文论证的一部分。每章重写时，同步对现有图表作出四种明确决定：

- **保留**：信息结构和正文位置都没有变化，只统一字体、线宽和图题；
- **重画**：核心关系仍然成立，但需要匹配 2.0 的案例、协议或视觉层级；
- **合并**：两张图表达的是同一条推理链，合成一张主图；
- **删除**：没有承担独立解释任务，或正文已经能够更清楚地说明。

### 1. 两套视觉语言

#### 架构与概念图

继续采用本书已经建立的“架构师白板手绘体”，但减少满图文字。手绘感主要用于：

- 责任边界；
- 控制流、数据流和证据流；
- 状态转换；
- 信任边界和故障传播；
- 架构演进与取舍。

手绘体只用于标题、节点名称和少量批注。较长说明统一使用清晰的正文黑体，避免整张图都像板书。

#### 表格、指标和评测图

数据表、状态映射、N-run 分布、SLO、成本和评测结果不使用随意的手绘比例，而采用出版级精确排版：

- 数值轴、单位、样本量和时间窗口完整；
- 示例数据与实测数据明确区分；
- 不用面积、透视或装饰图形夸大差异；
- 误差、方差和阈值在图中可见；
- 表格列数过多时拆分，不依赖横向滚动才能理解。

### 2. 统一视觉规范

- 可编辑源文件使用 SVG；PNG 只作为网站和兼容性输出。
- 所有图由版本化脚本生成，图中协议名称和版本可追溯。
- 采用暖纸背景、墨蓝主色和少量琥珀/绿色/红色强调色。
- 颜色不单独承担语义；同时使用线型、边框、图标或标签。
- 在普通笔记本宽度和移动端缩放后仍可读取，不在图中堆放正文段落。
- 每张图只回答一个核心问题，图题说明“这张图帮助读者判断什么”。
- 架构图必须标明边界、方向和责任，不使用只有圆圈与箭头的角色关系图。
- 状态机必须标出触发条件、非法转换和终止状态。
- 时序图必须区分请求、状态更新、Artifact 和副作用提交。
- 表格表头使用短语，单元格尽量使用完整但简短的判断，不放大段正文。
- 图像包含有意义的替代文本；正文不能只靠颜色或空间位置引用图中结论。
- 同时检查浅色、深色、灰度打印和 PDF 页面中的可读性。

每章原则上保留 4—5 张真正承担论证任务的主图。附加细节优先进入表格、代码或可交互示例，不追求“每章固定六张图”。

一张图只有满足以下条件才进入 2.0：

- 离开正文仍能看懂图的主题、方向和图例；
- 放回正文后确实减少了读者理解成本；
- 节点和箭头能够映射到正文中的真实合同或状态；
- 没有把厂商产品堆叠误写成架构；
- 协议字段、状态名称和版本与当前正文一致；
- 示例数字有来源或明确标注为演示数据；

- 图源脚本、可编辑文件、交付文件和 `visual-manifest.yml` 保持同步。

### 5. 排版同步调整

图表重构会同时带动正文排版：

- 章节开头不连续堆放引用、提示框和清单；
- 长表格拆分为正文判断、对照表和附录明细；
- 代码行宽适配网页和 PDF，不用横向滚动隐藏关键参数；
- 图题、表题、代码清单编号统一；
- 提示框只保留版本风险、安全警告和容易造成事故的关键事项；
- 章节页面的标题密度下降，正文段落获得更稳定的阅读节奏。

## 九、分阶段执行计划

### 阶段 0：出版样章和规则校准

先重写第 1 章，连同 CaseOps Slice 0、图表和 ADR 一起交付。通过样章确认：

- 第一人称语气是否自然；
- 理论深度是否合适；
- 代码和正文比例是否合适；
- 不同水平读者是否都能跟上；
- “去 AI 味”是否达到预期。

样章确认前，不批量重写后九章。

### 阶段 1：重写第一部分

重写第 1—3 章，完成从固定工作流到单 Agent、再到受约束多 Agent 的连续演进。

验收重点：

- 架构复杂度由失败和边界推动；
- CaseOps 能逐章运行；
- 工具、状态和协作合同一致；
- 不提前透支后续章节。

### 阶段 2：重写第二部分

重写第 4—6 章，完成 Context、平台和安全三条保证链。

验收重点：

- Context 与 Evidence 可追溯；
- 状态有所有者和恢复路径；
- 模型被操纵后仍不能越权；
- 安全和恢复都有可运行测试。

### 阶段 3：重写第三部分前半

重写第 7—9 章，完成系统合龙、评测与运营。

验收重点：

- A2A 1.0 与 MCP 当前规范准确；
- 协议状态与业务状态不混淆；
- Eval 可以定位责任层；
- AgentOps 能复盘真实复合故障。

### 阶段 4：重写第 10 章和全书前后文

完成第 10 章、序言、阅读建议、全书路线图、术语表和附录。

验收重点：

- 第 10 章引用的前九章产物真实存在；
- Clean-room 路径完整；
- GitHub 发布、SBOM、Provenance 和安全文件可验证；
- 全书首尾闭环。

### 阶段 5：全书编辑和技术终审

执行五类终审：

1. **结构审校**：章节递进、案例一致、交叉引用和重复内容。
2. **技术审校**：协议版本、代码、Schema、命令、标准成熟度和事实来源。
3. **运行验证**：单元、集成、评测、故障注入和干净环境复现。
4. **文字审校**：第一人称、句式节奏、术语、标点、图题和表题。
5. **发布审校**：MkDocs 构建、链接、移动端排版、图片清晰度和无障碍文本。

## 十、每章 Definition of Done

一章只有同时满足以下条件才算完成：

- 只围绕一个核心工程问题展开；
- 与前章存在明确因果连接，并为后章留下真实问题；
- 主案例保持同一业务对象、身份、权限和数据版本；
- 稳定原理、协议语义和参考实现明确分层；
- 至少包含一个简单方案、一个失败路径和一个工程化修正；
- 关键结论有官方资料、代码证据或明确推导支持；
- 代码可运行，命令经过验证，输出可复核；
- 图表与正文同步重构，回答明确问题，没有装饰性架构图；
- 图源、SVG、PNG、替代文本和视觉清单保持一致；
- 清单、表格和金句不过度替代连续论述；
- 没有虚构生产数据、客户经历或未经验证的行业结论；
- 读者能够完成一个本章实践并生成验收证据；
- 章节末尾能够自然解释下一章为什么出现。

## 十一、建议的启动方式

2.0 不从十章同时开工。建议先交付一个完整的第 1 章出版样章，其中包括：

1. 重写后的正文；
2. CaseOps Slice 0 可运行示例；
3. 一张新的架构判断图；
4. Agent Suitability Canvas；
5. ADR-001；
6. 编辑说明：1.0 哪些内容被保留、合并、删除或改写；
7. 技术和文字验收结果。

第 1 章确认后，再按照第 1—3、4—6、7—9、10 的批次推进。这样既能保持全书一致，又能让每一阶段都有完整、可审查的成果。
